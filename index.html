Cursor: 통합 엔진

VS Code를 기반으로 AI가 런타임에 내장된 IDE.

  • Cmd+K: 인라인 편집 엔진.
  • Composer: 다중 파일 생성 및 편집 에이전트.
  • Prediction: 다음 수정 사항을 예측하여 실행 ("Cpp").

Claude Code: 자율 분석가

주요 기능

Claude Code는 거대한 컨텍스트 윈도우를 가진 CLI 도구입니다.

  • 심층 추론: 전체 디렉토리 트리를 한 번에 분석 가능.
  • 도구 사용 (Tool Use): 테스트 실행, 에러 로그 확인, 코드 수정을 자율적으로 수행.
  • 아키텍처 우선: 단순 타이핑보다 '설계 및 계획'에 집중.

컨텍스트의 물리학 이론

토큰 윈도우 (Token Windows)

AI의 "작업 기억(Working Memory)" 용량.

함정: "Lost in the Middle". 컨텍스트가 가득 차면, 프롬프트 중간에 있는 세부 정보를 기억하는 능력이 저하됩니다.

RAG vs. Long Context

RAG (Cursor): 검색 및 추출. 빠르고 저렴하지만 의미적 연결을 놓칠 수 있음.

Long Context (Claude): 전체 읽기. 느리고 비용이 들지만 전체적인 이해도가 높음.

문서(Documentation)가 곧 API입니다

문서는 더 이상 사람이 가끔 읽기 위해 쓰는 것이 아닙니다.

문서는 에이전트가 실행할 "프롬프트 지시사항"입니다.

루트 컨텍스트 파일 (The Root Context File)

프로젝트에서 가장 중요한 단 하나의 파일입니다.

# Project: SaaS Platform

## Tech Stack
- Frontend: React 18, TypeScript, Tailwind
- Backend: Node.js, PostgreSQL
- State: Zustand

## Critical Rules
1. NO `any` types allowed. (any 타입 사용 금지)
2. Use functional components only. (함수형 컴포넌트만 사용)
3. All API calls must use `apiClient`. (모든 호출은 래퍼 사용)

실습 리뷰: 성공 기준

간결성 (Concise)

100줄 미만인가요? 컨텍스트 윈도우를 낭비하지 마세요.

구체성 (Specific)

정확한 버전을 명시했나요? (React 18 vs 16 차이 큼)

부정 제약 (Negative Constraints)

하지 말아야 할 것을 명확히 정의했나요?

스캐폴딩 전략

보일러플레이트를 직접 짜지 마세요. 아키텍트처럼 행동하세요.

에이전트에게 아키텍처 선호도를 참조하여 뼈대를 잡도록 시키세요.

// 스캐폴딩 프롬프트 예시

"Create a new React project. 
Structure the folders by Feature 
(Feature-Sliced Design). 
Initialize a `docs/` directory.
Setup `tsconfig` with strict mode."

"문서 우선(Doc-First)" 루프

1. 계획 (Plan)

자연어로 의도를 서술합니다.

2. 컨텍스트 (Context)

`docs/patterns/FEATURE.md` 작성.

3. 생성 (Generate)

문서를 참조하여 코드 구현.

4. 검증 (Verify)

패턴 준수 여부 리뷰.

1 & 2단계: 계획 및 컨텍스트

코딩을 멈추고, 명세하세요.

코드를 요청하기 전에, 해당 기능의 "물리학(법칙)"을 정의해야 합니다.

  • 상태(State)는 어디에 위치하는가?
  • 에러 처리는 어떻게 하는가?
  • 유효성 검사는 어떤 라이브러리를 쓰는가?

패턴 문서 해부학

패턴 문서는 AI 모델을 위한 학습 데이터(Training Data)입니다.

1. 의도 (Intent)

언제 이 패턴을 사용해야 하는가?

2. 정석 예시 (Canonical Example)

복사해서 쓸 수 있는 완벽한 코드 조각.

3. 안티 패턴 (Anti-Patterns)

명시적인 "DO NOT (하지 말 것)" 예시.

"정석 예시" (Canonical Example)

# Pattern: Data Fetching

## Canonical Example
export function useUser(id: string) {
  return useQuery({
    queryKey: ['user', id],
    queryFn: () => api.getUser(id),
    staleTime: 5 * 60 * 1000
  });
}

## Anti-Patterns
- DO NOT use `useEffect` for data fetching.
- (useEffect를 데이터 페칭에 사용하지 말 것)

이것은 모델에게 구체적인 예시를 제공하여 모방하게 하는 "Few-Shot Prompting" 기법입니다.

3단계: 기능 명세 (Feature Specs)

스코프된 컨텍스트 (Scoped Context)

특정 부분에만 적용되는 비즈니스 로직은 Tier 3 문서를 사용하세요.

예: 특정 폴더 내부의 AGENTS.md.

점진적 공개 (Progressive Disclosure)

"로그인 페이지"를 작업할 때 "결제 로직" 컨텍스트를 로드하지 마세요.

Tier 3 문서는 컨텍스트 윈도우를 깨끗하고 집중된 상태로 유지합니다.

실습 3: 문서 리팩토링 Hands On

과제: 환각(Hallucination) 수정하기

  • 트리거: 컴포넌트 생성을 요청했더니 인라인 스타일을 사용함 (나쁜 예).
  • 조치: 코드를 직접 고치지 마세요. docs/patterns/STYLING.md를 만드세요.
  • 재시도: "`docs/patterns/STYLING.md`를 참조해서 다시 만들어줘."
  • 결과: AI가 이제 Tailwind 유틸리티 클래스를 올바르게 사용합니다.

모듈 4: 협업 및 Git

코드로서의 컨텍스트 (Context as Code)

컨텍스트 문서는 코드베이스의 일부입니다. 버전 관리 시스템에 포함되어야 합니다.

  • 즉각적인 온보딩: 신규 입사자(및 그들의 에이전트)가 프로젝트 메모리를 즉시 상속받습니다.
  • 패턴 변경은 PR로: 아키텍처를 변경한다는 것은 PR을 통해 문서를 변경하는 것을 의미합니다.
  • 공유된 규칙: 모든 개발자의 에이전트가 동일한 표준을 준수하도록 보장합니다.

Context as Code (코드로서의 컨텍스트)

단일 진실 공급원 (SSOT)

컨텍스트 문서는 "위키 페이지"가 아닙니다. 소스 코드입니다.

반드시 다음을 따라야 합니다:

  • Git에 커밋됨.
  • 버전 관리됨.
  • Pull Request(PR)의 대상이 됨.

AI를 활용한 온보딩

과거의 방식

3주 동안 오래된 위키 읽기.

"저기 선배님, 에러 처리는 어떻게 하나요?"

AI 방식

즉각적인 온보딩.

신규 입사자 repo 클론 → Cursor 설치 → 에이전트가 `.cursorrules` 읽음.

"주니어 에이전트"가 이미 전문가가 되어 신규 입사자를 가이드합니다.

AI 보조 코드 리뷰

"신뢰하되 검증하라 (Trust but Verify)" 프로토콜

  • 패턴 준수: 문서와 일치하는가?
  • 환각 점검: 모든 import 구문을 확인하라.
  • 보안: 하드코딩된 비밀키가 없는가?

모듈 5

트러블슈팅 & 베스트 프랙티스

실패 유형과 고급 프롬프트 엔지니어링

일반적인 실패 유형

실패 유형 증상 해결책
컨텍스트 드리프트 코드는 변했는데, 문서는 그대로임. 문서 업데이트를 PR "완료 조건(DoD)"에 포함.
환각 (Hallucination) AI가 없는 라이브러리를 지어냄. Tier 1 문서에 "부정 제약" 추가.
게으른 코더 AI가 `// ...나머지 코드`라고 씀. 프롬프트: "코드를 생략하지 말고 전체를 작성해."

에이전트를 위한 프롬프트 구조

역할 (Role)

"시니어 아키텍트처럼 행동해."

작업 (Task)

"유저 폼을 만들어줘..."

컨텍스트 (Context)

"`docs/patterns/FORMS.md`를 참조해서..."

제약 (Constraint)

"...Zod를 사용하고 코드만 출력해."

생각의 사슬 (Chain of Thought)

"생각하는" 단계를 강제하세요

복잡한 작업의 경우, AI가 바로 코딩하게 두지 마세요.

먼저 계획을 세우게 하세요. 이는 "추론 토큰(Reasoning Tokens)"을 생성하여 최종 코드의 품질을 높입니다.

// CoT 프롬프트 예시

"Plan the database migration.
1. List potential risks.
2. Outline the steps.
3. Write the migration script.

Do not write code until the plan is approved."

# PROJECT_SNAPSHOT.md (상태 캐시 파일)

## High Level (개요)
- **Status**: Sprint 4 Active
- **Goal**: OAuth2 로그인 구현

## Current Context (현재 컨텍스트)
- **Active Module**: `src/auth` (인증 모듈 집중)
- **Recent Changes**: DB에 `user` 테이블 추가됨.

## Architecture Decisions (아키텍처 결정)
- 세션 관리에 `lucia-auth` 라이브러리 사용.

에이전트는 전체 저장소를 스캔하는 대신 이 단일 파일만 읽어 상황을 파악합니다.
